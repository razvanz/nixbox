# 003: Credential injection via explicit env var passthrough

**Date:** 2026-03-24
**Status:** accepted

## Problem

Credentials (GitHub tokens, AWS keys) must reach the guest VM without entering the Nix store or rootfs — both are world-readable and persist across rebuilds. The approach must also be explicit: users should declare exactly which credentials enter the sandbox, with no implicit sharing.

## Decision

Users declare credentials explicitly in the config using `builtins.getEnv` to pull host environment variables at eval time:

```nix
env = {
  GITHUB_TOKEN = builtins.getEnv "GITHUB_TOKEN_CLAUDE";
};
```

This config is resolved to JSON, then serialized onto an ephemeral ext4 disk (`env.img`, 32MB) at `nixbox up` time. The disk is hot-plugged via the cloud-hypervisor API as a virtio-blk device. A systemd oneshot (`inject-env`) copies `env` to `~/.env` (VM user's home) before `sshd` starts, then unmounts the disk.

What goes on the disk:
- **`env`** — key=value pairs from `env = {}`, sourced as `~/.env` at login

Tool-specific setup (git config, etc.) is handled by user-provided **scripts** that run after boot with env vars already sourced:

```nix
scripts = [
  "./scripts/setup-git.sh"
  "./scripts/setup-tools.sh"
];
```

Scripts are SCP'd into the guest and executed as the VM user. Relative paths resolve against the project directory.

The `builtins.getEnv` approach requires `--impure` for Nix eval, which is already required by the config resolution pipeline.

## Consequences

- **Explicit over implicit** — nothing enters the sandbox unless declared in `.nixbox/config.nix`. Adding a new credential is a one-line config change, not a code change.
- **No secrets in the Nix store** — `builtins.getEnv` resolves at eval time; the values never appear in derivations or the store.
- **Session-scoped** — the ext4 disk is destroyed on `nixbox down`. Credentials don't persist between sessions.
- **Re-auth without restart** — `nixbox aws login` SSH-injects fresh AWS credentials into a running VM, bypassing the disk entirely.
- **User scripts over built-in modules** — tool-specific config (git, SBT, etc.) is composed by users via `scripts`, not generated by nixbox. This decouples nixbox from any particular toolchain and lets users set up arbitrary tools using the same pattern.
